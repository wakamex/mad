02:13:50 claude:

Good question — this has scaling implications too.

The short answer: non-response is a no-op. The server doesn't process absent players.

Here's why that works:

No connection state exists. There are no WebSockets, no sessions, no "logged in" concept. The server only knows whether a player submitted an action for a given tick. No submission = no action taken. The server doesn't need to distinguish between "logged off," "crashed," "thinking too long," or "deliberately passing."

During batch evaluation, the evaluator iterates over the active_players list — only players who POSTed an action. Players not on the list are skipped entirely. If 200K of 1M players are active in a given tick, we evaluate 200K actions. The other 800K cost zero compute. This is ideal for scalability — the server's work scales with active players, not registered players.

What happens to absent players:

Tick type Non-response outcome
Standard/dossier 0 points. Missed opportunity, no penalty.
Interrupt/crisis Small bounded miss penalty. "You didn't dodge, so you got hit." But capped — a player who's offline for a crisis burst shouldn't hemorrhage unboundedly.
Quest-locked ticks No penalty. Player is expected to be unavailable for main-stream actions.

Side effects of absence: - Streak resets to 0. This is the main cost. Reconnecting means rebuilding your streak multiplier from scratch. Long absence = lost compounding. - Debt interest still accrues. Debt ticks happen on dossier cadence regardless of presence. If you have debt and go offline, it grows. This creates pressure to stay engaged if you're in debt, but doesn't punish debt-free players. - No aura change. You can't gain or lose aura while absent.

What this means strategically:

  • Optimal play is NOT 24/7 connection. It's selective engagement — be present for high-value ticks, skip low-value ones. But you can't predict which ticks are high-value without reading the stream, so in practice good players stay connected during active periods.
  • Casual players can dip in and out. Their score grows slowly when present and stagnates (or bleeds via debt interest) when absent. They're never catastrophically punished for having a life.
  • Random-baseline bots (the user's "rolling random selections from home" scenario) are WORSE than non-response. Random play with random confidence actively destroys score. Non-response just flatlines. This means "doing nothing" is literally a better strategy than "doing random things confidently" — which is thematically perfect for a benchmark about calibration.

Implementation in Go:

The batch evaluator is dead simple:

for _, playerID := range activePlayersThisTick {
    action := pendingActions[playerID]
    state  := playerStates[playerID]
    delta  := evaluate(action, tickPlan, state)
    applyDelta(state, delta)
}
// Separately, on dossier ticks only:
for _, state := range allPlayerStates {
    applyDebtInterest(state)
}

The first loop is O(active players). The second loop is O(all players) but runs only every 12th tick and does one multiply-and-add per player — trivial even for 1M.

One edge case worth noting: if a player submits an action but the POST arrives after the deadline (network latency, slow agent), the server rejects it with 410 Gone. From the player's perspective, they tried to act but didn't make it. This is scored as non-response, not as a wrong answer. The latency penalty is the missed opportunity, not a score penalty. This is fair because the deadline is known in advance from the tick's deadline_epoch.
